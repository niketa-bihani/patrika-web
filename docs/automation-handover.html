Executive summary

Where the framework stands today

The login test suite is fully operational and passing. It drives a real browser through the login flow on the live patrika.com site and validates the entry point, mobile-number handling, OTP triggering, and input validation — 19 automated checks in total.

The framework existed in draft form but could not execute: test runs hung indefinitely and, once unblocked, every test failed because the selectors did not match the real website. This session diagnosed the root causes, corrected them, and verified the suite green against production. The framework is now a reliable, repeatable regression asset.

Bottom line: from "will not run" to a stable 19/19 green suite, with the underlying causes fixed at the source rather than worked around.

What the framework is

Architecture & design approach

The framework follows the industry-standard Page Object Model (POM), separating what a test verifies from how the page is operated. This keeps tests readable and means a UI change is fixed in one place, not across dozens of tests.

LayerLocationResponsibility
Page Objectspages/Selectors & actions per screen — BasePage, HomePage, LoginPage
Test Suitestests/Business assertions, organised by scenario type
Utilitiesutils/CSV data loader & shared test helpers
Test Datatest-data/Test cases sourced from LoginTest.csv
Configurationconftest.py · pytest.iniBrowser fixtures, environment, markers, timeouts
Reportsreports/Self-contained HTML run report

Tests are data-driven: the four core cases (TC-LOGIN-01 to 04) are defined in a CSV that mirrors the manual test plan, so QA can extend coverage without touching code.

Engineering work completed

Three blocking defects — root cause & resolution

Defect 1 · Critical — suite would not run

Test runs hung indefinitely (event-loop deadlock)

The browser was created on a session-scoped async loop while each test ran on its own per-test loop. Every browser command then waited on a loop that never advanced — the run froze with no output, which looked like "tests not running."

Aligned the async loop scopes in pytest.ini so fixtures and tests share one event loop.

Defect 2 · High — navigation failed

Invalid-URL error on every test

Pages navigated to the relative path "/", but the browser context had no base URL configured, so Playwright rejected it as an invalid URL.

Set base_url on the browser context and removed a deprecated event-loop fixture.

Defect 3 · High — selectors did not match the site

Login selectors were assumptions, not the real DOM

The tests assumed a pop-up modal with English labels. The live site behaves differently — verified by inspecting the production DOM directly:

Tests assumedReality on patrika.com
Login opens a modalLogin is a separate /login page
English labels (Login, Get OTP)Hindi — लॉगिन, OTP प्राप्त करें, सत्यापित करें
One OTP input fieldSix single-character OTP boxes
"Get OTP" always clickableDisabled until a valid 10-digit number is entered

Rewrote all login selectors and flow logic to match the verified production DOM.

Hardening added: installed pytest-timeout so any future hang is auto-terminated at 30 seconds and reported as a failure — rather than freezing a run indefinitely, as the original defect did.

Known limits & recommended next steps

Honest scope today, and where to invest next

Current limitations

  • OTP verification is not end-to-end. A real code is sent to a physical phone, so tests validate up to "OTP field appears" — not a full logged-in session. A test number or mocked backend is needed to close this.
  • Negative tests assert client-side format logic rather than the site's server response. They are correct, but could be strengthened to assert the on-page error state directly.
  • Two config files (pytest.ini and pyproject.toml) define pytest settings; consolidating to one removes ambiguity.

Proposed roadmap

Near term

Close the OTP loop

Obtain a test mobile number or mock the OTP service to automate full login-success and post-login journeys.

Mid term

Broaden coverage

Extend Page Objects beyond login — home feed, search, article, subscription — reusing the same POM foundation.

Mid term

Cross-browser

Layer Firefox & WebKit onto the existing Chromium runs to catch rendering-specific defects.
